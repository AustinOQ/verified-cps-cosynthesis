RL_Phase Generalizability Analysis
===================================

This document describes what is general, what is mixing-model-specific,
how to expand the pipeline, and how to structure a SysML model so it is
compatible with the RL_Phase training pipeline.


1. What Is Fully General
------------------------

These components work for ANY SysML v2 model with a #Neural action def,
with zero modification:

  extractor.py
    Parses the SysML model, finds the single #Neural-annotated action def,
    and extracts observations (in params), actions (out params), the done
    binding expression (converted to Python), and observation bindings
    (which SysML ref each obs is wired to). This is pure SysML parsing
    with no domain assumptions.

  env.py — core loop (reset, step, reward, done, violation checking)
    - Observations are read from the SimulatorTwin state dict using the
      names from the #Neural action def. No name-matching or heuristics.
    - Actions are passed to the twin as a dict of {action_name: value}.
    - Done is evaluated by running the extracted done_expr as a Python
      expression with observation values substituted in.
    - Violations are checked via engine.requirement_statuses(), which
      returns {req_name: {kind, status}} for every requirement in the
      model. Only #Prohibition requirements with status=False trigger
      the -1 penalty and episode termination.
    - Reward: +1 (sparse) or +10 (shaped) for done, -1 for prohibition
      violation, 0 otherwise. #Obligation requirements are NOT penalized.

  env.py — distance-based reward shaping
    Parses the done_expr into a Python AST and computes continuous distance
    to the goal. This works for any done expression using numeric
    comparisons (>=, <=, ==) combined with and/or:

      (speed <= 0)                          -> max(0, speed)^2
      (x >= 10) and (y >= 20)              -> max(0,10-x)^2 + max(0,20-y)^2
      (temp >= setpoint) or (time >= limit) -> min(dist_temp, dist_time)
      (a - b) >= c                         -> max(0, c-(a-b))^2

    Per-step reward is the normalized improvement in distance:
      reward = (last_distance - current_distance) / initial_distance
    Total shaping reward over a successful episode sums to ~1.0.
    Goal bonus is +10 to keep the terminal reward dominant.

  train_rl.py
    SB3 PPO training with best-model checkpointing. No domain knowledge.
    Hyperparameters (n_steps=4096, target_kl=0.015, etc.) are tuned for
    CPS-scale problems generally, not the mixing model specifically.

  eval.py
    Deterministic evaluation with outcome breakdown (done/violation/timeout).
    Reports violation types and state traces for debugging.


2. What Is Specific to the Mixing Model
----------------------------------------

These components use naming conventions and patterns from the fluid-transfer
(mixing) model family. They work but are not general:

  env.py — _discover_mapping()
    Maps observation names to internal simulator engine state keys. This is
    needed because the SimulatorTwin's model_inputs dict uses observation
    names (e.g., "tank1VolumeMl") but the engine's internal state uses
    fully qualified keys (e.g., "system::feederTank1::currentLevelMl").
    To randomize start states, you must patch the ENGINE state, and there
    is no general API to do this.

    Current heuristics:
      - Looks for "currentLevelMl" suffix to find physical tank levels
      - Looks for "capacityMl" suffix to find tank capacity (upper bound)
      - Matches digits between obs names and engine keys (e.g., "1" in
        "tank1VolumeMl" matches "feederTank1" in the engine key)
      - Looks for "feeder" in engine keys to identify source tanks
      - Looks for "volumeSensor" to find sensor device readings
      - Looks for "controller" prefix in engine keys

    A brake system would have none of these patterns. Its engine state
    might have keys like "system::vehicle::currentSpeedKmh" and
    "system::brakeSystem::brakePressureBar".

  env.py — _randomize_start()
    Generates random valid start states. Classifies observations by role
    using naming conventions:
      - "volume"  = obs with discovered capacity (from _discover_mapping)
      - "target"  = obs with "target" or "transfer" in the name
      - "original" = obs with "original" in the name

    Then applies domain constraints:
      - Volumes randomized in [1, capacity]
      - Targets constrained to [1, paired_volume] (target <= volume)
      - Originals set equal to initial volume
      - Pairs observations by shared digits (tank1 <-> target1)

    Also updates derived booleans (isEmpty, isFull) in engine state.

    These constraints are correct for fluid transfer but meaningless for
    other domains. A brake system would need: speed in [0, maxSpeed],
    distance in [minDistance, maxDistance], etc.

    NOTE: The cross-observation constraints (target <= volume) are
    actually derivable from the done expression. If done says
    "(original - volume) >= target", then for the goal to be reachable,
    target must be <= original. This could be generalized by parsing
    the done_expr to infer valid ranges, rather than hardcoding names.

  env.py — done_threshold application
    The --done-threshold flag reduces observation values by a fixed amount
    before checking the done condition, but only for observations with
    "target" or "transfer" in the name. This naming assumption is not
    general. A more general approach would apply the threshold uniformly
    to all comparisons in the done expression.

  env.py — observation normalization with capacities
    When randomize=True and capacities are discovered, observations are
    normalized by their capacity values. When capacities aren't found
    (non-fluid models), it falls back to normalizing by the first
    episode's observation magnitudes, which is less stable across
    randomized episodes.


3. How to Make Randomization General
------------------------------------

The fundamental problem is: given an observation name, what is its valid
range, and what constraints link it to other observations?

Currently this is solved with naming heuristics. To make it general:

  Option A: Derive from done_expr (recommended)
    The done expression already encodes the goal relationships. Parse it
    to extract constraints:
      "(original - volume) >= target" implies target <= original
      "(speed <= 0)" implies speed starts > 0
      "(temp >= setpoint)" implies temp starts < setpoint

    Combine with range discovery from engine state (look for any key
    ending in a "capacity", "max", "min", "limit" pattern near the
    bound state keys).

  Option B: SysML model annotations
    Add range metadata directly to the #Neural action def:
      in speed : Real { range = [0, 200]; }
      in brakePressure : Real { range = [0, 1]; }
    The extractor would read these and pass them to the env.

  Option C: Per-model-family subclass
    Subclass SysMLEnv and override _discover_mapping() and
    _randomize_start() for each model family. Simple but not automatic.

  The obs-to-engine-state mapping (_discover_mapping) is the hardest
  part to generalize. The simulator's internal state representation
  is deeply nested with model-specific qualified names, and there is
  no API to say "set observation X to value Y" at the engine level.
  A general solution would require the SimulatorTwin to expose a
  set_observation(name, value) method that handles the internal mapping.


4. Expanding to Continuous / Multi-State Actions
------------------------------------------------

Currently all actions are Boolean (MultiBinary action space). The SysML
model already specifies action types (Boolean, Integer, Real) and the
extractor already extracts them — we just ignore them.

  Code changes (~30 lines in env.py):

    __init__:
      Instead of always using MultiBinary, check action_types:
        - All Boolean -> MultiBinary(n_act)           [current]
        - All Real    -> Box(low, high, shape=(n_act,))
        - Mixed       -> Dict or flattened Box space
      Bounds for Real actions need to come from somewhere (SysML
      constraints, model metadata, or reasonable defaults like [-1, 1]).

    step():
      Instead of bool(action[i]) for every action, convert based on type:
        - Boolean -> bool(action[i] > 0.5) or round
        - Real    -> float(action[i])  (possibly scaled)
        - Integer -> int(round(action[i]))

    SB3's PPO handles Box action spaces natively using Gaussian policies.
    No algorithm change needed. MlpPolicy works for both.

  Training impact:

    - Exploration: With Boolean actions, 2^n combinations (16 for n=4).
      PPO can stumble into the right one. With continuous actions, the
      space is infinite — reward shaping becomes essential, not optional.
    - Safety boundaries: Prohibition violations become continuous surfaces
      in action space rather than discrete bad combinations. The agent
      has more ways to barely violate or barely satisfy constraints.
    - Convergence: Gaussian policies explore via noise (high std early,
      low std late). May need more timesteps. The shaped reward signal
      still provides gradient toward the goal.

  System impact:

    - Verification (downstream): This is the real cost. With Boolean
      outputs, verification checks that no input maps to a bad binary
      combination. With continuous outputs, verification must prove
      properties over continuous output ranges — significantly harder
      for tools like beta-CROWN and nuXmv.
    - Simulator: Must handle continuous action values. The SimulatorTwin
      passes the action dict directly to the engine's model callback,
      so this should work if the SysML model's dynamics handle Real
      action values.

  Recommendation: Support it for generality but keep case studies Boolean
  unless the model genuinely requires continuous control. The RL training
  is straightforward; the verification cost is where it hurts.


5. How to Structure a SysML Model for Compatibility
----------------------------------------------------

For the pipeline to work out of the box, a SysML v2 model needs:

  A. Exactly one #Neural action def

    #Neural
    action def PolicyName {
        in observation1 : Real;
        in observation2 : Real;
        in done : Boolean;          // REQUIRED for goal detection
        out action1 : Boolean;
        out action2 : Boolean;
    }

    - in params = observations the agent sees each step
    - out params = actions the agent chooses each step
    - One in param must be named "done" (case-insensitive) with type
      Boolean. This is excluded from the observation vector and used
      for goal detection via its binding expression.
    - Currently out params must be Boolean. Real/Integer support is
      a ~30-line extension (see section 4).

  B. A SubactionCallStmt that invokes the #Neural action with bindings

    perform step : PolicyName {
        in observation1 = sensor1.response;
        in observation2 = controller.targetValue;
        in done = (observation1 >= threshold) and (observation2 <= limit);
        out action1;
        out action2;
    }

    - Each in binding maps an observation to a SysML expression or ref.
    - The done binding is the GOAL CONDITION. When it evaluates to True,
      the episode succeeds. This expression is also used for reward
      shaping (distance computation).
    - The done expression should use observation names as variables and
      numeric comparisons (>=, <=, ==, >, <) combined with and/or.
    - Other bindings record the ref path (e.g., "sensor1.response") so
      the env can discover simulator state key mappings.

  C. Requirement definitions with metadata annotations

    #Prohibition
    requirement def 'Safety Rule Name' {
        doc /* Human-readable description */
        require constraint { ... }
    }

    #Obligation
    requirement def 'Liveness Rule Name' {
        doc /* Human-readable description */
        require constraint { ... }
    }

    - #Prohibition: Violation causes -1 reward and immediate episode
      termination. These are hard safety constraints the agent must
      never violate. Examples: no dry running, no deadlock.
    - #Obligation: NOT penalized during training. The agent learns to
      satisfy these through the goal reward. Examples: must eventually
      complete, must eventually respond.

  D. Physical dynamics

    The model must have dynamics that the SimulationEngine can evaluate:
    state machines, flows, action step blocks, etc. The SimulatorTwin
    wraps the engine and advances the simulation each step.

  E. For randomization (optional, currently model-specific)

    If using --randomize, the model's internal state representation
    must be discoverable. Currently this uses heuristics (see section 2).
    For new model families, either:
    - Follow naming conventions that the heuristics can match
    - Subclass SysMLEnv and override _discover_mapping/_randomize_start
    - Wait for a general solution (section 3)

    Without --randomize, training uses the model's default initial state
    every episode. This works but produces a controller that may not
    generalize to other start states.


6. Summary
----------

  Fully general (any model):
    - Interface extraction
    - Environment core loop (step/reset/reward/done/violations)
    - Distance-based reward shaping from done expression
    - PPO training and evaluation

  Model-specific (mixing model heuristics):
    - Engine state key discovery (_discover_mapping)
    - Start state randomization (_randomize_start)
    - Done threshold application (name-matching "target"/"transfer")
    - Capacity-based observation normalization

  Straightforward extensions:
    - Continuous/multi-state actions (~30 lines, section 4)
    - General randomization via done_expr constraint inference (section 3)
    - SysML-annotated observation ranges (section 3, option B)
